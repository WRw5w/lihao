# 项目重构方案（保运行增量式）

> 原则：**每一步都是小步、行为不变、立刻验证、可即时回退**。任何时刻项目都处于可运行状态；所有文档里写过的命令（`python main.py`、`python exp_head.py --final ...` 等）全程保持原样可用。

**已验证能跑**：全部 6 个模块导入正常，`check_submission.py` 端到端 PASS。`config.py` 路径集中化已完成且自洽（main / exp_head / exp_round2 / finetune_lora / check_submission 的默认路径均已指向 config）。

- `outputs/lora/best.pt`（约 1 小时训练成本）
- `pred_results*.csv/zip`（已生成的提交文件）

## 一、目标结构

```
初赛数据集/
├── config.py                  # 不改名不挪动（所有脚本已依赖）
├── robustft/                  # 新增：核心包（纯库代码，无 argparse）
│   ├── __init__.py
│   ├── models.py              # CosineClassifier, LoRALinear, inject_lora, LoraClassifier
│   ├── data.py                # Dataset 类, transforms, LOAD_TRUNCATED_IMAGES
│   ├── denoise.py             # knn_agreement, fit_gmm_1d, per_class_topk_keep, 伪标签构造
│   ├── engine.py              # train_head, evaluate, per_sample_stats, seed, split
│   ├── features.py            # 特征提取与缓存读写/校验（从 main.py 抽出）
│   └── submission.py          # CSV/zip 生成 + 格式校验（三处重复合一）
├── main.py                    # 保留为薄壳入口（命令不变）
├── exp_head.py                # 薄壳 + re-export（命令与 from exp_head import X 均不变）
├── exp_round2.py              # 薄壳入口
├── finetune_lora.py           # 薄壳入口
├── check_submission.py        # 薄壳入口
├── tools/                     # 无关脚本移入（extract_cs.py 等）
├── docs/                      # 赛题文档、技术报告、EXPERIMENTS.md、本方案
├── tests/test_smoke.py        # 新增：分钟级冒烟测试
├── outputs/                   # 原样不动
├── train/  test/  *.zip       # 原样不动（git 排除）
├── requirements.txt           # 锁定实测版本
└── .gitignore
```

要点：**入口脚本留在根目录做薄壳**（内部只剩 argparse + 调包），而不是挪进 scripts/ 子目录——这是"命令一个字符都不变"的关键。

## 二、分步实施（每步含验证门与回退手段）

### Step 0：安全网（先做，零代码改动）

1. `git init` + `.gitignore`（排除 `train/`、`test/`、`*.zip`、`outputs/cache/`、`excel_head.json`），提交当前可运行状态作为基线 commit；
2. 记录基准指纹：`pred_results*.csv` 的哈希、`outputs/experiments*.json` 的关键数值；
3. 确立**金标准回归测试**（每步重构后必须复现）：
   - 导入检查：6 个模块全部可 import；
   - `check_submission.py` → PASS；
   - **数值回归**：固定 seed 重跑 exp_head 的 A_ce 配置（用特征缓存，约 1 分钟），noisy-val 必须精确等于 0.6264（epochs=15, seed=42）——这是行为未变的强证据。

> 回退手段：之后每步一个 commit，出问题 `git revert` 即回到能跑状态。

### Step 1：清理无关文件（零风险）

- `extract_cs.py`、`extract_excel_head.py`、`excel_head.json`、`计算机竞赛.md` → `tools/`（移动，不删除）；
- 赛题 pdf/md、技术报告、EXPERIMENTS.md → `docs/`；
- 验证门：导入检查 + check_submission PASS。

### Step 2：建核心包，exp_head.py 变薄（本次重构的主体）

1. 新建 `robustft/`，把 exp_head.py 中的共享代码**原样移动**（不改逻辑）到 models / denoise / engine；
2. exp_head.py 顶部改为 re-export：`from robustft.models import CosineClassifier` 等——**exp_round2.py 和 finetune_lora.py 的 `from exp_head import ...` 不需要任何改动**，老命令照常；
3. 注意事项：
   - `ImageFile.LOAD_TRUNCATED_IMAGES = True` 必须放在 `robustft/data.py` 模块级（Windows DataLoader worker 以 spawn 方式重新导入模块，设置必须随导入生效）；
   - Dataset 类移入包内对 Windows 多进程反而更稳（可导入路径更规范）；
   - checkpoint 兼容性安全：`best.pt` / `final_head_*.pt` 存的是 state_dict（纯张量）+ 普通 dict，类定义挪模块不影响加载。
- 验证门：导入检查 + **A_ce 数值回归 = 0.6264** + `finetune_lora.py --smoke`（约 3 分钟）跑通。

### Step 3：拆分 main.py

- 特征提取与缓存校验逻辑移入 `robustft/features.py`；旧基线训练逻辑保留在 main.py 薄壳中调用包内组件（或标注 deprecated 但保持可跑）；
- 验证门：`python main.py --max-train-samples 200 --max-test-samples 200 --work-dir outputs_tmp` 跑通（**务必用独立 work-dir，严禁指向 outputs/ 以免覆盖主缓存**），跑完删除 outputs_tmp。

### Step 4：统一提交生成

- 三处重复的 CSV/zip 逻辑合并到 `robustft/submission.py`，check_submission 的校验函数也移入，薄壳保留 CLI；
- 验证门：用 `outputs/lora/best.pt` 重新执行 `finetune_lora.py --predict`，新旧 `pred_results_lora.csv` 预测一致率应 ≥ 99.9%（同 checkpoint 确定性推理，正常应逐字节一致）+ check_submission PASS。

### Step 5：锁定环境与文档

- `requirements.txt` 锁实测版本（torch 2.11.0+cu130、timm 1.0.27、numpy、Pillow、tqdm、torchvision）；
- 重写 README：目录结构、全流程复现命令、各产物说明（半决赛"可复现代码 + 运行说明"交付物直接受益）；
- `config.py` 增加 `NUM_CLASSES = 500` 等少量常量（check_submission 已有硬编码 500），为复赛换数据集时只改 config 做准备。

### Step 6（可选）：tests/test_smoke.py

- 合成小张量上测 `knn_agreement`、`fit_gmm_1d`、`smooth_one_hot`、提交文件读写回环，秒级跑完；
- 价值：之后任何改动 30 秒内知道有没有砸坏核心函数。

## 三、明确不做的事（风险收益比不划算）

| 不做 | 原因 |
|---|---|
| 把项目挪出中文路径目录 | 当前一切正常（仅 traceback 显示乱码的美观问题）；挪动牵连缓存路径、IDE 配置、用户习惯 |
| 重命名/挪动 config.py、train/、test/、outputs/ | 全部脚本与缓存依赖这些位置 |
| 改特征缓存格式 | 缓存校验逻辑依赖现有 key（features/labels/class_names/image_names），改了就要重提 1.5 小时特征 |
| 引入 hydra/lightning 等框架级改造 | 比赛周期内收益小、引入回归风险大 |
| 删除任何现有文件 | 一律移动到 tools/ 或 docs/，不删 |

## 四、工作量与节奏

| 步骤 | 预估耗时 | 风险 |
|---|---|---|
| Step 0 安全网 | 15 分钟 | 无 |
| Step 1 清理 | 10 分钟 | 无 |
| Step 2 建包（主体） | 1–1.5 小时 | 低（有数值回归门） |
| Step 3 拆 main.py | 30–45 分钟 | 低 |
| Step 4 统一提交生成 | 30 分钟 | 低 |
| Step 5 环境与文档 | 30 分钟 | 无 |
| Step 6 冒烟测试 | 30 分钟（可选） | 无 |

总计约半天。建议在没有训练任务排队的窗口执行（重构期间不要并行跑训练，避免缓存/产物被并发读写）。
